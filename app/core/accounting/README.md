# Accounting Bounded Context

This Bounded Context (BC) manages the core business domain of the application: customers, invoicing, expenses, and legal audit trails.

Designed for **Mid/Senior Backend Engineer** interviews, this module applies opinionated, asymmetrical, and pragmatic architectural patterns.

---

## 1. Architectural Asymmetry (Rich vs. Flat Models)

Not all subdomains require the same level of abstraction. We apply the principle of **architectural asymmetry**:

- **`customers` & `expenses` (Flat Models)**: The logic here is relatively simple (CRUD or highly linear workflows). We have deliberately flattened these modules. Everything is centralized in a main service (e.g., `customer_service.ts`), backed by direct Drizzle queries (`queries.ts`). This reduces cognitive overhead.
- **`invoices` (Rich Model)**: The complex core of the BC. It requires a strict lifecycle (Draft → Issued → Paid), data snapshots (see below), and financial integrity calculations. This is the only subdomain organized in a more "academic" layered architecture (`domain/`, `application/`, `infrastructure/`).

---

## 2. The "Snapshot" Model (Invoices vs. Customers)

An `Invoice` is not just a database relationship to a `Customer`. It is a legal document frozen in time.

### How it works:

- **On Creation (Draft)**: The invoice workflow loads the current customer data (address, email, etc.). This data is **hard-copied** (snapshotted) into the invoice draft.
- **While Draft**: If a customer is updated, draft invoices refresh their customer snapshot so the operator can still correct customer details before issue.
- **On Issue**: The invoice is sealed (`status = 'issued'`).
- **Business Impact**: If the `Customer` address is updated later in the system, past invoices (already `issued`) retain the historical legal address. This progressive detachment is the cornerstone of our accounting integrity.

---

## 3. Observability and Audit Strategy

Observability is split into three distinct levels to ensure readability and legal security:

1.  **Request Logs (Technical)**: Standard logs generated by the request middleware. Useful for technical debugging and performance tracking (e.g., slow requests, 500 errors).
2.  **Security Logs (Attempts & Rejections)**: Structured traces of authorization failures (e.g., `403 Forbidden`, cross-tenant access attempts). These logs are crucial for "forensic readiness" and intrusion detection, but they do not pollute the business history.
3.  **Business Audit Trail (Legal)**: Stored immutably in the database (`audit_events`). It contains _only_ successful state transitions (e.g., "Invoice issued", "Member role changed"). This is the single source of truth for a financial auditor.

---

## 4. Degraded "Read-Only" Mode

The audit system (`audit_events`) is a critical dependency. If the audit database becomes unavailable or corrupted, the application does not crash with a fatal error.

Instead, it enters a **Degraded Read-Only Mode**:

- Read routes (`GET`) remain fully functional.
- The UI displays a clear warning (via the Inertia prop `accountingReadOnly`).
- Mutations (`POST`, `PUT`, `DELETE`) are intercepted and rejected with a clean technical error (`503 Service Unavailable`).

This ensures that the business does not suffer a total loss of consultation capabilities while protecting the absolute integrity of accounting entries (we do not record money if we cannot audit it).

---

## 5. Multi-Tenant Isolation (Structural Security)

The application is fundamentally multi-tenant (SaaS). Isolation does not rely solely on the goodwill of HTTP controllers.

- **Defense in Depth**: The `tenantId` (or `organizationId`) is systematically enforced at the boundaries (middlewares) and explicitly passed down into execution contexts (`AccountingAccessContext`).
- **Schema Constraints**: The database reinforces these invariants with composite foreign keys and `CHECK` constraints to make it impossible to insert orphaned cross-tenant rows (e.g., linking a tenant A invoice to a tenant B customer).
- **Access vs. Provenance**: `organizationId` is the authorization boundary. `createdBy` and audit `actorId` are provenance metadata and may be nullable for system/bootstrap actions.
- **Active Membership Gate**: HTTP routes must only construct accounting access after the session's active organization has been validated against an active `member` row.
